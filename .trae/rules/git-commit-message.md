# 素缃 · AI 提交信息生成规则

> 素缃一卷，墨香千年。
> AI 提交，意境悠然。

## 🎋 提交信息生成规范

### 语言风格要求
- **典雅文言**：使用「之」、「其」、「乃」等文言词汇
- **诗意表达**：融入传统意象和诗词元素  
- **意境营造**：每个提交都应如诗如画
- **简洁明了**：主题不超过50字，正文每行不超过72字

### 提交类型映射
根据 git diff 内容自动选择诗意类型：

| 变更类型 | 诗意类型 | 意象 | 诗意描述 |
|----------|----------|------|----------|
| 新增功能 | **新章** | 开卷 | 如开新卷，添锦绣之章 |
| 修复错误 | **修误** | 补墨 | 如补墨迹，使字迹清晰 |
| 文档更新 | **文墨** | 添注 | 如添注释，增文采之韵 |
| 代码格式 | **雅致** | 调色 | 如调布局，求和谐之美 |
| 代码重构 | **重构** | 重装 | 如重装帧，使结构更佳 |
| 测试相关 | **校验** | 验色 | 如验墨色，保质量之优 |
| 构建工具 | **杂务** | 理房 | 如理文房，备笔墨纸砚 |

## 🏮 主题生成规则

### 主题应如诗题
- **动词开头**：使用「新增」、「优化」、「调整」等
- **意境深远**：如「朝霞映纸」、「暮色渐浓」
- **简洁明了**：如「素缃初成」、「墨色微调」

### 主题生成示例
- 新增功能 → **新章: 素缃·朝霞主题初成**
- 修复错误 → **修误: 补墨迹对比之误**
- 文档更新 → **文墨: 添素缃意境之章**
- 代码格式 → **雅致: 调素缃布局之韵**

## 📜 正文生成规则

### 正文如诗序
- **描述变更**：如「宣纸底色由浅黄调为米白」
- **说明原因**：如「为增强阅读舒适度」
- **意境营造**：如「让代码如诗如画」

### 正文生成模板
根据 git diff 内容生成诗意描述：

```
- <变更描述1>，<意境效果1>
- <变更描述2>，<意境效果2>  
- <变更描述3>，<意境效果3>

<诗词收尾>
```

## 🖋️ 页脚生成规则

### 页脚如落款
- **关联Issue**：如 `Closes #12`
- **Breaking changes**：如 `BREAKING CHANGE: 重构配色体系`
- **诗词收尾**：如「素缃一卷，意境悠长」

```
- 添加四象配色哲学说明，美学更佳
- 完善安装指南文言表达，文采更浓

文墨添彩，意境更浓。
```

## 🎋 生成算法规则

### 类型识别算法
1. 分析 git diff 中的文件变更类型
2. 根据变更内容匹配诗意类型
3. 优先选择最符合变更内容的类型

### 主题生成算法  
1. 提取变更的核心功能或问题
2. 转化为诗意表达的主题
3. 确保主题简洁明了，意境深远

### 正文生成算法
1. 逐条分析具体变更内容
2. 为每条变更添加诗意描述
3. 使用诗词收尾营造意境

## 🏮 特殊场景处理

### 多类型变更
当提交包含多种类型变更时：
- 优先选择主要变更类型
- 在正文中说明其他次要变更
- 保持整体诗意一致性

### 复杂变更
对于复杂的技术变更：
- 简化技术描述为诗意表达
- 重点突出变更的意境效果
- 避免过于技术化的语言
